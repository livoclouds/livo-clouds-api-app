Phase 6 — Architecture decision

Three pagination models were evaluated before any code change. Option A was selected.

Option A — Server-side pagination by resident range (chosen)

DimensionDetail

Decision: Option A

  • Bounds the payload (the explicit Phase 6 goal in performance-analysis.md:78-87).
  • Mirrors Phases 4 & 5 patterns exactly — same { data, meta } envelope, same DTO style, same defaults strategy.
  • No live UI consumer: fetchCollectionYear and fetchCollectionMatrix have zero callers; the matrix UI consumes MOCK_COLLECTION_RECORDS. Lockstep cost is minimal: type-only on the web side.
  • limit=500 (matrix) and limit=600 (flat collection) preserve today's behavior for every current tenant (largest seed < 300 residents).
  • Matrix correctness is trivially preserved — see the equivalence proof below.

Matrix equivalence proof

Let M(year) be the full logical matrix for a tenant: rows are residents ordered by unitNumber asc; columns are 12 months.

  • Single-page case (limit ≥ |R|): new response has data.length === |R|, same order, identical per-row content. JSON.stringify(oldResponse) === JSON.stringify(newResponse.data).
  • Multi-page case: M = concat(page1.data, page2.data, …) over pages ordered by unitNumber asc. Each page.data is a contiguous slice because skip/take preserves order.
  • Per-row month payload: months[] bounded at 12 entries by construction (where: { year }, orderBy: { month: 'asc' }).
  • Flat endpoint /collection: same logic — data is a row-paginated slice ordered by month asc.

Manual snapshot validation is deferred to post-deploy because the working environment has no live API at planning time. Documented commands: capture before.json from current main, deploy Phase 6, capture after.json with limit=10000, then jq '.data' after.json | diff - <(jq '.' before.json).
