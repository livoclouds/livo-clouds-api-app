Executive summary

The API is structurally sound. Tenant isolation, authentication, and Prisma index coverage are all in good shape. The dominant risks are payload size from ten unbounded list endpoints and a handful of in-memory aggregations that should be SQL. None of these would break the system today, but they will all degrade together as tenant data volume grows. See README.md for navigation.

Top critical risks

  • Residents list unbounded P1.1
  • Collection year matrix unbounded P1.2
  • Resident account statement unbounded P1.3
  • Overdue report unbounded P1.4
  • Collection matrix report unbounded P1.5

Top high-impact opportunities

  • Dashboard trend → SQL aggregate P2.1
  • classifyBatch batched UPDATE P2.2
  • Imports upload parallelism P3.1
  • Calendar list require date range P3.4

Acceptable as-is — do not change

  • Tenant isolation guard R1
  • Response & error envelopes R3.1 R3.3
  • Index coverage on hot models
  • Append-only transaction flow
  • Soft-delete pattern

Top performance findings

Each finding cites file:line in the API repo. Full detail in performance-analysis.md.

critical P1.1 Residents list returns full roster with 3-level include

residents.service.ts:15-25 — findMany({ include: { vehicles, pets, additionalResidents } }) with no take. Payload grows linearly with tenant size; today acceptable, will not survive a 500-unit tenant.

API+web (rolling) fix later · Phase 5

critical P1.2 / P1.5 Collection matrix endpoints unbounded

collection.service.ts:16 and reports.service.ts:31 — return all residents × 12 months per request. Today ~3,600 rows for a 300-unit tenant; doubles each fiscal year if multi-year matrix is added.

API+web (lockstep) fix later · Phase 6

critical P1.3 Resident account statement loads every transaction in memory

collection.service.ts:35-117 — all transactions for a resident, all collection records, then totals via JS .reduce. No date-range default, no pagination.

API+web (rolling) fix later · Phase 5

critical P1.4 Overdue report unbounded with nested includes

reports.service.ts:8-29 — all overdue residents + nested unpaid collection records, then .map(). Grows with delinquent base and overdue months.

API+web (rolling) fix later · Phase 5

high P2.1 Dashboard trend builds Map<month, Set> in JavaScript

dashboard.service.ts:84-153 — loads every paid collection record for the year, builds a per-month set of residentIds in JS. Equivalent SQL is a single COUNT(DISTINCT) query.

API-only fix later · Phase 1

high P2.2 classifyBatch issues per-row UPDATE in chunks of 200

classification.service.ts:389-477 — 200 concurrent prisma.transaction.update(...) per chunk. A 1,000-row import = 1,000 round-trips. Bounded by connection-pool size.

API-only fix later · Phase 3

medium P3.1 Imports upload performs sequential I/O per file

imports.service.ts:78-191 — for each of up to 5 files: hash, dedup lookup, optional delete, create, R2 upload, all serialized. Multiple console.log statements in production code.

API-only fix later · Phase 1

medium P3.4 Calendar list has optional date range

calendar.service.ts:32-71 — without from/to returns every non-deleted event for the condominium. Web always sends a range today; the API does not require it.

API-only fix later · Phase 2

Top risk findings (non-performance)

Tenant isolation, authorization, contract stability, data consistency, and operational risks. Full detail in risk-analysis.md.

medium R3.2 Pagination response shape is inconsistent across modules

Transactions / imports return a flat shape ({ data, total, page, ... }); audit returns an envelope ({ data, meta: {...} }). The declared PaginatedResult type matches the envelope but services drift from it.

API+web (lockstep) fix later · Phase 4

medium R5.3 R2 upload failures are silently swallowed

imports.service.ts:177-179 — storage failure is logged but the batch is still returned as "queued". User has no signal that the bank statement file was not retained.

API+web (lockstep) fix later · Phase 5

medium R5.2 File buffering up to 100 MB per request

Fastify multipart configured for 20 MB × 5 files; SHA-256 is computed over the in-memory buffer. Memory pressure under concurrent uploads.

API-only fix later · Phase 1 (stretch)

low R4.1 Petty-cash folio uses non-atomic count+1

DB constraint @@unique([condominiumId, folio]) prevents corruption — but concurrent writes produce a P2002 error rather than a retried success. UX issue, not a data issue.

API-only fix later · Phase 1

acceptable R1.x · R6 Tenant isolation is solid — do not change without dedicated plan

condominium-access.guard.ts:12-53 — validates slug → resolves condominiumId → enforces session match (ROOT bypass). Bulk reconcile and manual match both re-verify ownership.

no action acceptable as-is
